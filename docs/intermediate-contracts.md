# Intermediate Contracts

The project is a staged pipeline. Each stage should write files a human can inspect and, eventually, correct.

## PdfInspection

Produced by:

```powershell
python -m score2gp.cli inspect-pdf input.pdf --out work/inspect
```

Current file:

```text
work/inspect/inspect_pdf.json
```

Purpose:

- classify the PDF as born-digital, raster/scanned, or mixed
- render page images
- record page dimensions
- extract text blocks and bounding boxes when available
- preserve enough data for diagnostic overlays

Status: implemented as a basic diagnostic contract.

## TabRaw

Produced by:

```powershell
python -m score2gp.cli extract-tab input.pdf --out work/tab
```

Current file:

```text
work/tab/tab_raw.json
```

Purpose:

- hold candidate fret numbers, chord symbols, and technique text
- preserve source page and bounding boxes
- preserve stable candidate IDs
- preserve system/staff/bar/string estimates when known
- preserve x/y evidence for later MusicXML alignment
- keep confidence values visible
- keep uncertain text as candidates rather than forced notes

Status: `tabraw.v0.1` candidate contract implemented. Candidate IDs must be unique. Extraction is still first-pass born-digital text candidate collection only; `build-ir` currently consumes fret candidates and reports chord-symbol or technique-text candidates as preserved but not aligned.

The controlled public fixture `tests/fixtures/pdf/generated_tiny_tab.pdf` is the first real PDF producer test for this contract. It is generated by `tests/fixtures/pdf/make_generated_tiny_tab_pdf.py` using PyMuPDF and contains simple vector tab lines, barlines, and positioned fret text.

The score-like public fixture `tests/fixtures/pdf/generated_scorelike_tab.pdf` is generated by `tests/fixtures/pdf/make_generated_scorelike_tab_pdf.py`. It keeps the same public, hand-generated nature, but adds two tab systems, two bars per system, spacing variation, chord symbols, technique text, candidate text, a chord-like vertical stack, and multi-digit frets. `extract-tab` can infer string and global bar estimates for these fixtures by combining word boxes with detected six-line tab systems and vector barlines.

The uneven-spacing public fixture `tests/fixtures/pdf/generated_uneven_spacing_tab.pdf` is generated by `tests/fixtures/pdf/make_generated_uneven_spacing_tab_pdf.py`. It deliberately includes one visually well-spaced bar and one risky bar where playable x groups sit too close together. This fixture exists to test diagnostics, not to make alignment more permissive.

`extract-tab --out` accepts either a directory, which writes `tab_raw.json`, or a `.json` file path for smoke tests:

```powershell
python -m score2gp.cli extract-tab tests/fixtures/pdf/generated_tiny_tab.pdf --out work/generated_pdf/generated_tiny_tab.tabraw.json
python -m score2gp.cli extract-tab tests/fixtures/pdf/generated_scorelike_tab.pdf --out work/generated_pdf/generated_scorelike_tab.tabraw.json
python -m score2gp.cli extract-tab tests/fixtures/pdf/generated_uneven_spacing_tab.pdf --out work/generated_pdf/generated_uneven_spacing_tab.tabraw.json
```

## MusicXmlImport

Produced by:

```powershell
python -m score2gp.cli omr input.pdf --out work/omr --audiveris "C:\Program Files\Audiveris\Audiveris.exe"
```

Current files:

```text
work/omr/*.mxl
work/omr/*.omr
work/omr/audiveris.log
work/omr/warnings.json
```

Purpose:

- use Audiveris for standard notation OMR
- capture logs and warnings
- parse uncompressed partwise MusicXML into limited timing, pitch, voice, rest, tie, tuplet, harmony, and selected guitar technique data
- provide timing input for the first synthetic ScoreIR builder

Status: Audiveris wrapper implemented. A limited MusicXML parser is implemented for synthetic fixtures and supports plain `.musicxml`/`.xml` plus native `.mxl` packages. Compressed `.mxl` parsing reads the declared rootfile from `META-INF/container.xml` without extracting to disk. Repeat expansion, alternate endings, grace timing, and full MusicXML semantics are not implemented yet.

## ScoreIR

Intended produced by:

```powershell
python -m score2gp.cli build-ir --musicxml work/omr/score.musicxml --tabraw work/tab/tab_raw.json --out score.ir.json
```

Purpose:

- normalize recognised notation and tablature into a strict score contract
- store tracks, tunings, bars, timing, notes, techniques, confidence, and provenance
- be suitable for validation, semantic comparison, correction, and writing

Status: ScoreIR v0.1 schema and validation are implemented. `build-ir` supports narrow synthetic MusicXML + TabRaw paths and controlled generated-PDF TabRaw paths. Real private-score alignment is still pending.

Current synthetic alignment details:

- MusicXML provides bar timing, rests, notated duration, tuplets, chord symbols, and selected note techniques.
- TabRaw provides string/fret candidates, consumed by bar and x-position order.
- Pitched ScoreIR events are emitted only when TabRaw provides string/fret evidence.
- Unused fret candidates, non-fret TabRaw candidates, unattached harmonies, and pitch mismatches are explicit warnings.
- PDF-derived chord symbols, technique text, and unknown candidate text are preserved as non-playable evidence and are not consumed as notes.

Optional sidecar diagnostics:

```powershell
python -m score2gp.cli build-ir --musicxml tests/fixtures/musicxml/tiny_multibar.musicxml --tabraw tests/fixtures/tabraw/tiny_multibar_tabraw.json --out work/synthetic/multibar.ir.json --diagnostics-out work/synthetic/multibar.diagnostics.json
```

The sidecar uses `build-ir-diagnostics.v0.1` and includes imported event counts, loaded/matched/unmatched TabRaw candidate counts, unmatched MusicXML counts, unsupported warning codes, confidence flags, and per-bar alignment summaries. It is intentionally separate from ScoreIR so diagnostics can evolve without changing the interchange contract.

For PDF-derived TabRaw, diagnostics also include counts for fret/non-fret/chord-symbol/technique/unknown candidates, candidates with bbox/x/y/system/string/bar evidence, ignored non-playable candidates, source-stage counts, per-system summaries, and extraction quality flags such as missing inferred string/bar, multiple inferred tab systems, or low-confidence candidates.

Per-bar x-to-onset diagnostics compare only playable fret candidates against MusicXML pitched onset groups. They report candidate x groups, MusicXML onset groups, relative visual and musical positions, drift/error values, chord-stack detection, close-group ambiguity, and a quality label: `good`, `warning`, `poor`, or `unknown`. Chord symbols, technique text, and candidate text remain non-playable evidence; they are counted and warned, but they are not included in x-to-onset calibration and are not consumed as notes.

## ConversionReport

Produced by:

```powershell
python -m score2gp.cli convert --pdf input.pdf --template fixtures/templates/minimal_gp7.gp --out output.gp --work-dir work/run1
```

Current files:

```text
work/run1/warnings.json
work/run1/conversion-report.html
```

Purpose:

- summarize recognised items
- list guessed items
- list unsupported items
- preserve warnings rather than hiding uncertainty

Status: basic scaffold implemented.

## Contract Rule

Every stage should either read/write ScoreIR or read/write a clearly documented derivative listed here. Recognition uncertainty belongs in the data, not in console-only output.
