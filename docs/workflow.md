# Workflow

This document describes the intended staged workflow from an owned PDF score to a Guitar Pro `.gp` package.

The full PDF-to-GP workflow is not complete yet. The current project can inspect PDFs, run Audiveris, parse a limited MusicXML subset from `.musicxml`, `.xml`, or native `.mxl` packages, build ScoreIR from synthetic MusicXML plus TabRaw fixtures, run controlled generated-PDF extraction/alignment smoke tests with x-to-onset diagnostics, inspect/write minimal GP packages, and validate generated packages. Private real-world PDF-derived MusicXML/tab alignment is still pending.

## 4. Extract Tablature Candidates

Extract born-digital text candidates for fret numbers, chord symbols, and technique labels:

```powershell
python -m score2gp.cli extract-tab "fixtures/private/Derek Trucks BB King.pdf" `
  --out "work/derek/tab"
```

Outputs:

```text
work/derek/tab/tab_raw.json
work/derek/tab/warnings.json
work/derek/tab/grouping-diagnostics.html
work/derek/tab/overlays/page-001-grouping.png
work/derek/tab/inspect/inspect_pdf.json
work/derek/tab/inspect/pages/*.png
```

Current status: first-pass candidate extraction only. It writes `tabraw.v0.1` candidates with stable IDs, bounding boxes, x/y evidence, confidence, and nullable staff/string/bar estimates. For born-digital PDFs with vector six-line tab systems and visible barlines, it can infer string and bar estimates heuristically. It does not yet perform full tab staff/string/beat alignment.

When extraction finds playable-looking fret text, `extract-tab` writes `grouping-diagnostics.html` and one overlay image per page. Open the HTML report first. It records whether grouping is `grouped`, `partial`, or `missing`; whether alignment was attempted; and whether ScoreIR was written. The overlays show extracted token boxes plus any inferred tab staff boxes, six string lines, barlines, bar boxes, and assigned string/bar labels. Missing or partial grouping means the evidence is inspectable but unsafe for automatic `build-ir`.

PDF grouping v0.1 is intentionally narrow. It uses born-digital drawing geometry from the public generated fixtures: six near-horizontal tab lines define a tab staff, vertical lines crossing that staff define bar boxes, fret text is assigned to the nearest string line, and x-position chooses the bar. It does not perform OCR, recover hidden staff geometry, or calibrate arbitrary engraving. For the future design and constraints regarding automatic layout recovery of partial grouping failure modes, see the [PDF Layout Partial-to-Recovery Strategy](partial-to-recovery.md) design note.

These fixtures are generated by `tests/fixtures/pdf/make_new_layout_diagnostics_pdfs.py`. They test the system segmentation boundaries and warning taxonomy:
- **Candidate Outside System**: Fret candidate is placed horizontally outside the drawn tab lines system boundary.
- **Candidate Outside Bar**: Fret candidate is placed outside system barlines.
- **Multi-System Order Ambiguous**: Two horizontal tab systems vertically overlap/interleave. The greedy gap-cluster algorithm detects them but flags them as ambiguous.
- **ASCII & Drawn Layout Conflict**: Page contains both a drawn vector tab staff and raw ASCII tab rows. Detectors run separately, conflict is flagged, preventing conflation.
- **Prose & Legend Text**: Page has legend and prose text only (no drawn tab lines or systems). The detector correctly identifies `pdf_no_systems_detected`, `pdf_tab_staff_missing`, and `pdf_string_lines_missing` even with zero playable fret candidates.
- **Text & Geometry Present but No Safe System**: Drawn geometry (arbitrary lines) and text are present but do not form a safe tab system staff. Correctly flags `pdf_text_geometry_present_but_no_safe_system` and `pdf_drawn_geometry_present_but_staff_unresolved`.
- **Tab Candidates Present but System Not Detected**: Playable candidates exist, but no staff systems are detected. Correctly flags `pdf_tab_candidates_present_but_system_not_detected`.
- **Tab Staff Lines Fragmented**: Staff lines exist but are fragmented (shorter than the horizontal line threshold). Correctly flags `pdf_tab_staff_lines_fragmented`.
- **Candidates Between Systems**: A fret candidate is placed vertically between two valid systems. Correctly leaves the candidate unassigned to system/string and blocks `build_ir`.
- **Candidates Unassigned to String**: A candidate is placed within a system box but too far from any of the six string lines, flagging `pdf_candidates_unassigned_to_string`.
- **System Order Ambiguous Close**: Two systems are placed too close vertically, flagging vertical overlap order ambiguity.
- **Mixed Prose & Tab-Like Numbers**: Page has normal prose/legend text containing numbers (like "1", "2") with no staff lines. Playable candidates are extracted but unassigned, ensuring they never become trusted playable grouping and block `build_ir`.
- **ASCII Tab Three Blocks No Bars**: Page contains three ASCII tab blocks with no usable vertical bar separators. Correctly flags `pdf_ascii_system_detected` and `pdf_ascii_system_measure_boundaries_missing`.
- **System Detected No Barlines**: A drawn six-line system exists on the page, but no vertical barlines or bar boxes are detected. Correctly flags `pdf_system_detected_bar_detection_missing` and `pdf_input_class_drawn_tab_requires_barlines`, indicating that system detection succeeded but bar detection is the next blocker stage.
- **Valid Grouped Counterpart**: A drawn six-line system with barlines, representing a valid grouped page. Used to verify the successful counterpart flow where both system and bar detection succeed.
- **Barlines Below Threshold Crossing Staff**: A compact system (spacing = 6.0pt) where barlines are 30pt (below 40pt threshold) but cross all six lines. Correctly accepted by relative crossing rules.
- **Barlines Below Threshold Crossing Partial Staff**: Barlines are short (25pt) and do not cross the staff, flagging `pdf_barline_does_not_cross_staff` and rejecting.
- **Barlines Above Threshold Outside Staff Region**: Vertical lines are tall (45pt) but placed entirely above the staff region, flagging `pdf_barline_outside_staff_region` and rejecting.
- **Barlines Crossing Top Bottom Missing Middle**: Barlines cross the top and bottom lines but are missing lines in the middle (insufficient gaps crossed), flagging `pdf_barline_crosses_insufficient_string_gaps` and rejecting.
- **Barlines Crossing All Gaps Short Absolute**: A compact tab system where barlines are 30pt (below 40pt) but cross all gaps. Accepted.
- **Barlines Crossing Only Some Gaps**: Barlines cross only 2 gaps, flagging `pdf_barline_crosses_insufficient_string_gaps` and rejecting.
- **Compact Barlines Safe Boxes**: A compact system with valid barlines that successfully constructs bar boxes and groups.
- **Compact Barlines Candidate Outside**: Compact valid barlines succeed, but a candidate fret number is outside all bars, flagging `pdf_candidate_outside_bar` and blocking downstream.

These fixtures are generated by `tests/fixtures/pdf/make_fret_refinement_pdfs.py` and test our fret-number refinement and grouping diagnostics:
- **Clean Single-digit**: Valid staves with single-digit fret numbers centered on strings.
- **Clean Multi-digit**: Valid staves with multi-digit fret numbers like 10, 12, 15 horizontally merged.
- **Split Span Merged**: Horizontal merge of adjacent digits written very close (gap = 3.0 pt <= 5.0 pt).
- **Gap Too Large**: horizontal merge is rejected because the gap is too large (gap = 9.0 pt > 5.0 pt).
- **Vertical Misalignment**: horizontal merge is rejected because digits are vertically misaligned (dy = 3.0 pt > 2.0 pt).
- **Technique Marker**: Mixed technique words (e.g. `7h9`, `5/7`, `8b`) are proportionally split, retaining digits as fret candidates and technique letters as non-playable technique markers.
- **Chord/Section Heading Exclusions**: Digits in chord symbols or section text above the staff (`A7`, `Verse 2`) are kept as non-playable and excluded.
- **Page/Legend Exclusions**: Digit-like text outside system candidate bounds (`Legend: 1 = Ring`) are kept as non-playable and excluded.
- **Oversized Tall**: Fret number candidate whose bounding box height is too tall.
- **Tiny Noisy**: Candidate whose bounding box is too small or noisy, capping its confidence.
- **Grouped Success**: Successfully grouped counterpart verifying the E2E flow.

All these warning classes prevent `build_ir` from compiling ScoreIR, keeping grouping safety gates strictly intact.

These fixtures are generated by `tests/fixtures/pdf/make_edge_boundary_fallback_pdfs.py` and test our edge-boundary fallback diagnostics:
- **Edge Left Fallback**: Edge system with one right boundary successfully infers left boundary using clear system edge.
- **Edge Right Fallback**: Edge system with one left boundary successfully infers right boundary using clear system edge.
- **Edge Ambiguous Fallback**: Inference is rejected because of a rejected barline candidate in the inference direction.
- **Edge Too Narrow Fallback**: Inference is rejected because the inferred bar box would have a width less than 30pt.
- **Edge Candidate Near Inferred**: Inference is rejected because a playable fret candidate is located too close to the inferred boundary.
- **Non-Edge Internal Missing**: Proves that missing internal (inter-measure) boundaries are never inferred and remain strictly unsafe.
- **Multi-System Safe Fallback**: Multi-system page where one system successfully uses safe edge fallback and all playable candidates assign cleanly, completing grouping.
- **Multi-System Partial Fallback**: Multi-system page where a fallback rejection keeps the page in partial grouping status.
- **Next Blocker String Assignment**: Safe fallback succeeds, but a candidate too far from any string triggers string assignment blockers downstream.
- **Empty System Policy Fallback**: Decorative/empty systems are safely ignored, while active systems use safe edge fallback to group successfully.

All these warning classes prevent `build_ir` from compiling ScoreIR, ensuring that unsafe groupings are strictly blocked.

They model born-digital text PDFs where tablature is encoded as ASCII rows rather than drawn staff geometry. `extract-tab` uses `ascii-tab.v0.1` to group six-row ASCII blocks, assign row-order strings, extract fret tokens, estimate token x-position from character spans, and preserve inline technique markers as non-playable evidence.

The ASCII timing contract is `ascii-timing.v0.1`. It records column indexes, normalized column positions, aligned bar-separator columns, measure segment IDs where available, and timing status. Bar separators can make measure segmentation inspectable, but ASCII character position is not musical timing by itself. No-bar fixtures emit `ascii_tab_timing_unavailable` and `ascii_tab_measure_boundary_missing`. Barred/equal-width fixtures emit `partial_ascii_tab_timing`. Uneven fixtures emit `ambiguous_ascii_tab_timing`. Malformed fewer-than-six-row blocks emit `partial_ascii_tab_grouping`. `build-ir` refuses all of these before ScoreIR output unless a future phase supplies safe MusicXML/ASCII alignment semantics.

This path is intentionally tiny. The `ascii-scoreir-gate.v0.1` diagnostics report `ascii_scoreir_gate_status`, a primary reason code, secondary reason codes, candidate counts, aligned candidate counts, rejected candidate counts, alignment sidecar status, MusicXML timing safety, output event count, whether ScoreIR was written, and an expected remediation hint. It only allows the controlled public monophonic fixture when MusicXML timing is safe, every playable ASCII fret candidate has compatible alignment evidence, strings/frets come from TabRaw, and durations/rests come from MusicXML. Missing sidecars, partial/ambiguous/incompatible/unavailable alignment, unsupported techniques, chord/symbol requirements, polyphony, tuplets, ties, grace notes, missing string/fret evidence, missing MusicXML duration evidence, or risky MusicXML timing still block ScoreIR output.

Refusal diagnostics are useful outcomes. Common categories include `missing_ascii_alignment_sidecar`, `ascii_alignment_status_unavailable`, `ascii_alignment_status_partial`, `ascii_alignment_status_ambiguous`, `ascii_alignment_status_incompatible`, `ascii_alignment_candidate_missing`, `ascii_alignment_not_one_to_one`, `ascii_candidate_missing_string`, `ascii_candidate_missing_fret`, `ascii_candidate_unmapped_measure`, `ascii_candidate_unmapped_onset`, `ascii_unsupported_technique_required`, `ascii_unsupported_chord_symbol`, `ascii_polyphony_not_supported`, `ascii_musicxml_timing_risk`, `ascii_duration_source_missing`, and `ascii_outside_tiny_gate_scope`. When `--diagnostics-out` is supplied, `build-ir` writes these in the `build-ir-failure-diagnostics.v0.1` sidecar rather than writing unsafe ScoreIR. In addition, when it refuses due to ASCII ScoreIR gate diagnostics, it writes a developer-facing HTML diagnostics report (`ascii-scoreir-gate-diagnostics.html`) alongside the JSON sidecar to explain the refusal clearly to developers. The HTML report is developer-facing only; JSON remains the programmatic source of truth. Refusal is the expected behavior for unsupported ASCII inputs, and this HTML report does not imply broader ASCII-to-ScoreIR support.

Current status: implemented for limited synthetic fixtures and controlled generated-PDF fixtures. It uses MusicXML for measure timing, rests, voices, chords, backup/forward timing, chord symbols, tuplets, and selected note techniques. It accepts plain MusicXML/XML and compressed MXL packages by reading `META-INF/container.xml` and the declared rootfile directly from the zip package. It uses TabRaw for string/fret candidates and simple bar/x-order alignment. It emits warnings for unused candidates, non-fret TabRaw candidates that are preserved but not aligned, missing tab evidence, and pitch mismatches. The optional diagnostics file reports imported, matched, unmatched, ignored non-playable, extraction-quality, per-system, per-bar, and x-to-onset summaries. It uses standard guitar tuning as an explicit placeholder. Private real-world fixture alignment is still deferred.

When symbol/technique attachment diagnostics exist, `build-ir` writes a developer-facing HTML inspection report (`symbol-attachment-diagnostics.html`) alongside the JSON diagnostics to assist developers in reviewing attached and unattached chord symbols and technique text evidence.
- **Developer Inspection**: The HTML report provides a human-readable visualization of which candidates successfully attached (with their target bar indexes, event IDs, and note string/fret details) and which remained unattached (with their confidence values and warning codes).
- **JSON as Source of Truth**: The JSON diagnostics payload remains the programmatic source of truth.
- **Scope Limits**: GPIF technique/chord rendering remains out of scope. Symbols and techniques cannot create timing or notes; they are only attached as conservative metadata/evidence to existing, safely timed events. Unsupported or ambiguous text remains strictly diagnostic evidence (warnings) rather than being promoted or silently dropped.

`build-ir` now preflights MusicXML timing using a deterministic voice cursor/timeline model before writing ScoreIR. Public Audiveris-like fixtures under `tests/fixtures/musicxml/` cover overfull bars, chord stacks, and backup/forward timing movement. The preflight simulates voice cursor tracking per-measure, per-part, and per-voice. Precise timing preflight diagnostics distinguish the following categories:

- `valid_compound_meter`: Compound meter (e.g. 12/8) measure with correct timing extents and no errors.
- `musicxml_voice_timeline_valid`: The voice timelines are completely valid.
- `musicxml_voice_duration_underfull`: A voice cursor timeline is underfull compared to the expected measure duration.
- `musicxml_voice_duration_overfull`: A voice cursor timeline exceeds the expected measure duration.
- `musicxml_same_voice_measure_overfull`: A voice cursor timeline exceeds the expected measure duration.
- `musicxml_same_voice_event_overlap`: Overlapping note/rest durations within the same voice timeline.
- `musicxml_same_voice_rest_note_overlap`: Rest and note overlaps within the same voice timeline.
- `musicxml_event_extends_past_measure`: An event's duration extends beyond the measure end.
- `musicxml_accumulated_duration_overflow`: Accumulated small rounding errors lead to measure duration overflow.
- `musicxml_invalid_duration_grid`: The expected duration divisions do not partition into a clean integer grid.
- `musicxml_timing_repair_not_attempted`: Flag showing that timing repair was not performed.
- `musicxml_timing_calibration_required`: Timing calibration/repair is required for alignment to succeed.
- `musicxml_measure_duration_inconsistent_across_voices`: Active voice timelines have inconsistent end ticks across the measure.
- `musicxml_backup_cursor_before_measure_start`: A `<backup>` element attempts to rewind the voice cursor before the start of the measure.
- `musicxml_forward_cursor_after_measure_end`: A `<forward>` element pushes the voice cursor beyond the expected measure duration.
- `musicxml_same_voice_overlap`: Overlapping active note durations within the same voice.
- `musicxml_cross_voice_overlap_unsupported` / `musicxml_valid_multivoice_unsupported`: Overlapping durations between different voices (unsupported cross-voice polyphony).
- `musicxml_chord_stack_detected` / `musicxml_chord_stack_not_timing_overlap`: Distinctly classifies valid note stacks/chords encoded with `<chord/>` so they are not treated as unsafe overlaps.
- `musicxml_chord_stack_without_anchor`: A `<chord/>` element appears without a preceding anchor note in that voice.
- `musicxml_rest_overlap_same_voice`: Rest and note overlaps within the same voice timeline.
- `musicxml_repeated_backup_forward_risk`: Measure has 4 or more backup/forward cursor movements, exceeding the safe limit of 3.
- `musicxml_many_timing_risks`: Measure has high-density timing risks with more than 5 errors or highly overlapping event density.
- `musicxml_voice_cursor_alignment_risk`: Appended to cross-voice timing overlap, rest overlap, or same-voice overlap if backup/forward movements are present.
- `musicxml_alignment_not_attempted_due_to_timing_risk`: Appended as a final blocker issue if any error exists, refusing ScoreIR generation.


Overfull bars or same-voice overlaps are treated as timing risk and refused before invalid ScoreIR is written. Unsupported polyphony (valid multi-voice timing that build-ir cannot represent) is explicitly refused as `musicxml_scoreir_polyphony_gate_refused` rather than a timing risk:

```powershell
python -m score2gp.cli build-ir `
  --musicxml "tests/fixtures/musicxml/audiveris_like_overfull_bar.musicxml" `
  --tabraw "tests/fixtures/tabraw/tiny_single_bar_tabraw.json" `
  --out "work/timing/overfull.ir.json" `
  --diagnostics-out "work/timing/overfull.diagnostics.json"
```

This command is expected to exit non-zero and write a `build-ir-failure-diagnostics.v0.1` payload when `--diagnostics-out` is supplied. That is intentional: the project should refuse known-invalid timing rather than rely on a later generic ScoreIR validation error. In addition, `build-ir` automatically generates a developer-facing HTML diagnostics report (`musicxml-timing-diagnostics.html`) beside the JSON diagnostics file to provide an inspectable visual table of timing issues, affected measures/voices, expected/actual ticks, per-voice duration totals, and remediation hints. Just like other HTML reports, it serves as a human-readable visual inspection helper while JSON remains the programmatic source of truth.

`build-ir` also refuses PDF-derived TabRaw when playable fret candidates exist but system/string/bar grouping is absent. In that case the failure category is `missing_pdf_grouping`, and no ScoreIR file is written. This keeps ungrouped fret text from being treated as reliable musical evidence. ASCII-tab inputs are also refused until timing is safe; complete ASCII-row extraction fails with `ascii_tab_timing_unavailable`, while incomplete ASCII row grouping fails with `partial_ascii_tab_grouping`.

Timing refinement v1.0 adds explicit JSON classifications without changing conversion gates. For MusicXML failures, inspect `musicxml_timing_refinement.timing_classification`: `invalid_timing_refused` means the MusicXML timeline is broken, while `unsupported_polyphony_refused` means timing was valid enough to classify but ScoreIR/build-ir does not yet support that multi-voice shape. `automatic_repair_attempted` remains `false`.

When reading `build-ir-diagnostics.v0.1`, check each bar's `quality`, `mapping_quality_classification`, `candidate_x_groups`, `musicxml_onset_groups`, `mean_absolute_relative_error`, `max_relative_error`, `ambiguous_x_group_count`, and `x_to_onset_warnings`. `safe` means the current x-order heuristic is internally consistent for the controlled input. `partial`, `ambiguous`, `incompatible`, `refused`, or `unknown` should block trust in automatic conversion until the geometry and MusicXML timing are inspected.
